# 데모 텔레메트리 활성화 가이드 (#168) — 디렉터 체크리스트

> 대상: mhdmj. 코드는 전부 들어가 있고(v1.9.4-demo), **아래 값 2개를 채우기 전까지는 아무것도 수집·발송하지 않습니다**(네트워크 요청 0).
> 파일 위치: `src/telemetry.js` 상단 — `EP`(GA4 키)와 `STEAM_STORE_URL`(위시리스트 버튼).

## 0. 지금 상태 (아무것도 안 해도 되는 상태)

- 이벤트 4종은 플레이어 기기의 로컬 큐에만 쌓입니다. 밖으로 나가는 요청은 없습니다.
- 위시리스트 버튼은 크레딧 마지막 카드에 심어져 있지만, 스토어 URL이 빈 값이라 **숨김**입니다(죽은 버튼 방지).
- 수집 항목은 익명입니다: 설치 단위 랜덤 id + 이벤트 이름 + CTA 문안 배정값(variant). 계정·기기·IP 저장 없음(GA4가 IP를 좌표화하지 않도록 아래 3-c에서 끕니다).

## 1. GA4 쪽에서 할 일 (1회, ~10분)

1. https://analytics.google.com 접속 → 구글 계정으로 로그인.
2. **관리(톱니) → 계정 만들기** → 계정 이름 `Nine Winters`.
3. **속성 만들기** → 속성 이름 `Nine Winters Demo`, 시간대 `대한민국`, 통화 아무거나.
4. 플랫폼 선택에서 **웹** 선택 → URL은 아무거나(예: `https://coketown.github.io`) — 우리는 gtag를 안 쓰고 서버 프로토콜(Measurement Protocol)로 직접 쏘므로 URL은 형식용입니다.
5. 만들어진 **데이터 스트림** 클릭 → 여기서 값 2개를 챙깁니다:
   - **측정 ID** (`G-XXXXXXXXXX` 형태) → `measurementId`
   - 같은 화면 아래 **Measurement Protocol API 비밀번호(API secrets)** → 「만들기」 → 별명 아무거나 → 생성된 **비밀번호 값** → `apiSecret`

## 2. 코드에 값 넣기 (1분 — 나한테 값만 주셔도 됩니다)

`G:\pw-demo\src\telemetry.js` 상단:

```js
const EP = { measurementId: 'G-XXXXXXXXXX', apiSecret: '여기에' };
export const STEAM_STORE_URL = 'https://store.steampowered.com/app/<앱ID>/';  // 스토어 페이지 살아나면
```

- **주의**: apiSecret은 클라이언트에 실리는 값입니다. 게임 텔레메트리 관례상 수용되는 리스크(누가 알아내도 할 수 있는 건 우리 통계에 쓰레기 이벤트를 넣는 것뿐)지만, 결제·계정 같은 데 쓰는 비밀번호와 절대 겹치지 않게 하십시오.
- 값이 들어가는 커밋에 **옵트아웃 설정 토글**(설정 → 게임플레이 → "익명 사용 통계 보내기", ko/en/ja)을 제가 동봉합니다 — 수집이 실제로 시작되는 시점과 옵트아웃 제공 시점을 일치시키는 게 원칙이라, 지금 미리 넣지 않고 묶어 뒀습니다.
- 이후 재빌드+재릴리스하면 그 빌드부터 수집 시작.

## 3. GA4 설정 마무리 (수집 시작 후 1회)

- a. **관리 → 데이터 보관**: 14개월로 늘려두기(기본 2개월).
- b. **보고서 → 실시간**: 데모 실행하고 `demo_start`가 몇 분 안에 뜨는지 확인 — 이게 뜨면 파이프 전체가 산 겁니다.
- c. **관리 → 데이터 수집**: 구글 신호 데이터 수집 OFF 유지(우리는 광고 연동 불필요), 세분화된 위치·기기 데이터 수집 OFF.

## 4. 뭘 보게 되나 (퍼널 읽는 법)

| 이벤트 | 뜻 | 보는 법 |
|---|---|---|
| `demo_start` | 새 게임 시작(설치 단위 1회) | 분모 — 전체 시작 수 |
| `demo_complete` | 15일 컷 도달 | **완주율** = complete ÷ start |
| `credits_reached` | 크레딧 실제 시청 | 완주 후 이탈(모달에 묻힘 등) 점검 |
| `wishlist_click` | 위시리스트 버튼 클릭 | **전환율** = click ÷ credits — D2 CTA A/B의 심판 |

- 모든 이벤트에 `variant` 파라미터(`lack`=결핍 정본 / `hope`=희망 변형)가 실립니다. 지금은 전원이 결핍 카피를 보므로 variant는 배정만 기록됩니다 — **희망 카피 문안을 결재해 주시면** 노출 분기를 올려 실제 A/B가 시작됩니다(QUALITY-UP.md §2 D2).
- 탐색: GA4 **탐색(Explore) → 유입경로 탐색(Funnel exploration)** → 위 4이벤트를 순서대로 단계에 넣고, 분류(breakdown)에 `variant`.

## 5. 요약 — 디렉터가 실제로 할 일

1. GA4 계정/속성 만들고 **측정 ID + API 비밀번호** 2개를 나에게 전달 (§1)
2. 스팀 스토어 페이지가 살아나면 **스토어 URL** 전달
3. (수집 시작 후) 실시간 보고서에서 demo_start 확인 (§3-b)
4. (선택) 희망 CTA 카피 결재 → A/B 시작
