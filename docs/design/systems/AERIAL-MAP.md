## ★ 개정 2차 (2026-07-22 밤 — 디렉터 원안 재설명. 결정 지점 확정 + 시안 지위 강등)

디렉터 원문 요지: **"시안은 시안일 뿐이야."** 원안 재설명 —
1. **overview** = 항공 위성지도 같은 **라이브 화면**에서 어디로 탐사할지 고른다.
2. **노드(각 파밍 지점) 클릭** → 그 장소의 **실시간 근접 화면**(레퍼런스 2번 느낌).
3. 근접 화면 위에 정보 UI: **어떤 아이템이 파밍 가능한지 · 확률 · 들고갈 장비(준비물) 등** → **출발 결정은 여기(focus)서**.

**개정 1차와의 차이**: 정보 카드·출발 CTA의 자리가 overview 측면 패널이 아니라 **focus 근접 뷰**다. overview는 "고르는 화면", focus가 "판단하고 떠나는 화면". 현행 `startExpedition` 준비 모달(성공률 분해·준비물 선택·확인)의 내용이 focus 위 정보 UI로 승격되는 구조 — 데이터·플로우는 기존 그대로, 무대만 바뀐다.

**시안 지위**: 두 GPT 시안(탐색 지도/SHELTER 거점)은 **무드 레퍼런스일 뿐 스펙이 아니다**. 시안의 무시스템 요소(인원·산소·시설 내구도·아이템 핫바·다중 경유 「경로에 추가」)는 접수하지 않는다. 노드 명칭도 픽션 — 실 지역명 사용.

**UI 상세 규약**: 디렉터 고지 — "UI 적 부분은 지금 문서가 잘못된 것도 있다. 작업 시 재설명하겠으니 **질문하라**." → focus 정보 UI·HUD 연동 등 UI 구체 설계는 문서를 근거로 확정하지 말고 **착수 시점에 디렉터 질문 경유**가 규약이다.

## ★ 개정 3차 (2026-07-22 밤 — 진입 프레임·측면 패널 확정 + 진입 연출 논점)

디렉터 3차 입력(군용 터미널 레퍼런스 — 사진 UI 자체는 참고 금지, 프레임만):
- **프레임 확정**: "터미널을 통해 위성사진과 그에 대한 디테일한 정보를 찾고, **진입한다**" — 지도는 조회 화면이 아니라 **위성 콘솔 세션**이다.
- **측면 패널 확정**: 정보·장비는 좌 또는 우 **사이드 패널**로 따로 보인다(중앙=위성 뷰 점유). focus에서 **측면 바로 들고갈 장비(준비물) 선택**, **back으로 overview(전체 위성 지도) 복귀**. → 앞선 질문 2(장비 선택 자리)의 답: 현행 탐험 준비 모달의 내용이 focus 측면 패널로 이식된다.
- **열린 논점〔D〕**: 탐험 진입 연출 — ⓐ 전체 화면 즉시(몰입) vs ⓑ PDA가 나오며 화면이 확대되는 트랜지션. "별도의 Pip-Boy류 기기는 없으니까" — 콘솔 픽션을 어디에 둘지의 문제.

**Fable 분석(권고 ⓐ+터미널 부팅 트랜지션)** — 판단 근거 5:
1. **PDA 물성 보존**: 우리 PDA의 확립된 픽션은 손바닥 LCD(도크 53px·7~9px 폰트·베젤)다. 이걸 풀스크린으로 늘리는 모프는 "폰이 벽이 되는" 문제 — Pip-Boy는 처음부터 풀스크린 기기로 설계된 픽션이고 우리 PDA는 아니다. 확대 트랜지션이 오히려 기기 픽션을 깬다.
2. **몰입의 실체는 의식(ritual)**: 기하학적 모프가 아니라 **콘솔 점유의 부팅 시퀀스**(CRT 스윕+플리커+`// UPLINK` 타이프라이터, 0.3~0.5s)가 "위성 회선에 접속했다"를 말한다. MU/TH/UR 콘솔 호출 문법과 정합.
3. **실면적**: 라이브 디오라마+측면 패널은 폭이 필요 — 모달/PDA 베젤 안에서는 사이드 패널이 안 나온다. 모바일서 패널을 시트로 접는 여지도 풀스크린이라야 생긴다.
4. **렌더 예산**: 풀스크린 = 본편 씬 완전 스로틀(개정 1차 기왕 규약), 전환 경계가 깨끗. PDA 모프는 도크→풀스크린을 해상도×uiz×모바일 매트릭스로 검증해야 — 넥페 역산에서 리스크.
5. **레지스터 분리**: 휴대(PDA)=요약·GPS·앱, **콘솔(풀스크린)=위성 작전·출발 결정**. 두 레지스터가 겹치지 않고 서로를 살린다. PDA 지도 탭의 「전도 열기」는 위성 콘솔의 진입점으로 유지.

구현 귀결(결정 무관 공통부): 지도는 모달이 아니라 **전용 풀스크린 레이어**(#satmap-screen, title/journal-screen 전례)가 된다 — ⓐ든 ⓑ든 종착 뷰포트는 풀스크린이라 컨테이너 작업은 공통, 트랜지션만 갈아끼운다.

## ★ 개정 1차 (2026-07-22 — 디렉터 확정 3건, 초판 §0·§2·§5를 대체)

디렉터 레퍼런스 2차(GPT 생성 예시) 판독: ①**탐색 지도** = 도시 항공 실사 위 노드 그래프(노드·연결선·경로 점선·선택 노드 카드: 위험도 해골·이동시간·주요 수집품·날씨, 하단 탐색/경로/노드정보/확대/**출발**) ②노드 클릭 → **그곳의 항공뷰 라이브**(눈 내림·난로 불빛·시간대 반영) → **여기서 출발을 확정**.

1. **실시간 확정** — "날씨나 해가 지는 정도에 따라 달라지는" 라이브 렌더. 초판의 베이크(B안) 1차 채택은 **폐기**, 베이크는 lowSpec 폴백으로만 강등.
2. **데모(Next Fest 10월) 필수 탑재.**
3. **최종 판단은 실 모델링+픽셀화 프로토타입으로** — 디렉터가 실물을 보고 결정. 따라서 1단계 산출물은 프레임워크가 아니라 **판단용 프로토타입**이다.

### 개정 아키텍처 — "단일 라이브 디오라마 씬 + 카메라 2상태"

초판의 "전도=이미지, 근접=별도" 이원 구조를 버린다. **도시 디오라마 씬 하나**를 라이브로 돌리고, 카메라가 두 상태를 오간다:

```
[overview] 원경 항공(도시 전체) ──노드 클릭: 카메라 돌리 줌──▶ [focus] 노드 근접 항공
     ▲            노드 그래프 오버레이(DOM, 3D→화면 투영)              │ 출발 CTA
     └───────────────────── 뒤로/ESC ──────────────────────────────────┘
```

- **왜 단일 씬인가**: 전도↔근접이 같은 씬이면 줌이 **연속 카메라 이동**이라 레퍼런스 1→2의 체험이 그대로 나오고, 날씨·시간 동기화도 한 곳이다. 씬 2벌 관리·전환 로딩이 사라져 이원 구조보다 오히려 단순하다.
- **디테일 예산**: 도시 = 저폴리 블록 인스턴싱(InstancedMesh — ruin-city 프리팹 재사용) + 안개·어둠 + **본편과 동일 픽셀화 포스트**. 픽셀화가 저디테일을 가려주는 게 이 룩의 핵심 배당 — "실제 모델링과 픽셀화에 따라 결정"이라는 디렉터 판단 기준과 일치. 노드 반경만 드레싱 밀도를 올린다(focus에서 가까이 보이는 곳만).
- **실시간 동기화**: 씬은 본편 상태(gameHour·weather.type·계절·적설)를 **구독**한다. 요구 선행: `applyTimeLighting`/날씨 적용을 전역 의존에서 **(scene, ctx) 인자형으로 추출** — [REFACTOR-GUIDE](../eng/REFACTOR-GUIDE.md) **P1(렌더 컨텍스트)의 부분 착수가 이 기능의 전제**다. 지도가 P1을 견인한다.
- **렌더 자원**: 지도 열림 동안만 두 번째 소형 렌더러(내부 해상도 ~640, 픽셀화 업스케일) 생성, 닫으면 dispose(soak 검증 패턴 존재). 본편 렌더는 지도 열림 중 스로틀. 모바일 = 내부 해상도 캡, lowSpec = 현행 2D 지도 폴백 유지.
- **노드 오버레이**: 핀·라벨·경로선은 DOM(글리프·둥근모 재사용) — 매 프레임 3D 월드좌표→화면 투영으로 위치 갱신. 초판 자료구조(§3 mapnodes 월드좌표 SSOT)는 그대로 유효하며, **투영이 베이크 타임이 아니라 런타임**이 될 뿐이다. `mapnodes.gen.json`은 불요로 폐기.
- **정보 카드**(레퍼런스 우측): 위험도·이동시간·수집품·날씨 — 전부 기존 데이터(REGIONS rate/risk·시그니처·WEATHERS)의 재조판. 신규 시스템 0.

### 3.2 베이크 산출 — `src/data/mapnodes.gen.json` (생성물, 수정 금지)

베이크 도구가 월드좌표를 항공 카메라로 **투영해 픽셀 % 좌표를 자동 산출**한다. 현행 "% 수작업 튜닝"이 사라지는 지점 — 지형(디오라마)이 바뀌면 재베이크만 하면 마커가 따라온다.

```json
{ "home": { "residential": { "px": 41.2, "py": 33.8 }, "slum": { "px": 22.7, "py": 63.1 } } }
```

state: { level: 'city'|'focus', city, focusId }

- `focus`에서 `#map-info`(성공률 분해)·`#map-pull`은 그대로 산다 — 정보 소유권 불변(상세는 PDA, 지도는 판단).

## 6. 검증 계획

- **베이크 결정론**: bake-aerial은 시드 고정 + freezeForGolden 문법 — 같은 입력 = 같은 PNG(diff 0). 재베이크가 곧 회귀 테스트.
- **골든 확장**: 지도 모달(전도)·focus(근접) 2씬을 골든에 추가. 단 **라이브 디오라마(P2)는 #212 크로스프로세스 함정 지대** — lodge 플레이키와 같은 부류가 재발할 수 있으니, P2 착수 전에 REFACTOR-GUIDE **P1(렌더 컨텍스트 객체)** 선행을 강권한다. 정적 베이크(P0·P1)는 이 리스크가 0이라 계단 순서가 안전하다.
- **좌표 정합**: 투영 산출 % vs 실제 클릭 히트 — 배터리에 "전 노드 px∈안전영역 + 노드 간 최소 간격" 게이트 추가.
- **모바일**: 2048 텍스처 1장 + 640 컷 1장 동시 로드가 최대치(레벨 전환 시 이전 해제). 위젯 모드는 지도 자체가 안 뜨므로 무관.
