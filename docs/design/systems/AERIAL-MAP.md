# 항공뷰 지도 — 설계서 (AERIAL-MAP)

> **오더**(디렉터 2026-07-22): 절차 생성 이미지 지도를 **항공뷰 시점의 모델링**으로 대체하고, 노드 클릭 시 **그 지점의 근접 환경**을 보여준다(레퍼런스: 어두운 항공 지도 → 아이소 근접 디오라마). 목적 = **"내가 어딜 정확히 탐사하는지"의 공간 실감**. 레퍼런스는 예시일 뿐, 구현 가능성·파일 설계·자료구조를 설계하라.
> **지위: 설계서 — 구현 미착수.** 관련: [../ui/UI-REWORK-REVIEW.md](../ui/UI-REWORK-REVIEW.md)(지도 상세는 PDA 소유) · [../eng/REFACTOR-GUIDE.md](../eng/REFACTOR-GUIDE.md)(관문 원칙) · GD-2.0 §9.8(도시뷰).

## 0. 판정 요약

**구현 가능하다 — 신규 엔진 기능이 아니라 기존 자산의 재배치다.** 이 저장소에는 이미 ①오프스크린 하네스(골든·트레일러 캡처 — 3D 씬을 결정론적으로 이미지화) ②셸터 16종의 옥외 환경 빌더(`buildEnv`) ③폐허 도시 프리팹(ruin-city 빌딩·#206 동부 전도) ④지역·도시 데이터 테이블이 있다. 없는 것은 **도시 항공 디오라마 1~2장, 노드 좌표의 단일 소스, 베이크 도구, 줌 전환 연출** 4가지뿐이다.

핵심 제안 한 줄: **항공 전도는 "실시간 3D"가 아니라 "3D를 구워낸 이미지"로 시작한다** — 이 게임의 문법(방치형·저사양 위젯·APK 147MB·오프스크린 결정론 QA)에 맞고, 근접 뷰만 나중에 라이브 3D로 승격하는 계단을 둔다.

## 1. 현행 실측 (무엇을 대체하는가)

| 요소 | 현행 | 파일:줄 |
|---|---|---|
| 지도 배경 | 절차 비오메 캔버스 → dataURL (`mapBiomeDataUrl(city)`, 서부/동부 분기) | game.js:3089 (동부 전도 3240, #206 맨해튼 리워크) |
| 노드(핀) | DOM div, `MAP_MARKERS[rid]` **% 좌표** + 안전영역 클램프 | game.js:2937~ |
| 도시 탭 | `citiesReached` 기반 칩 (#198-f) | game.js:2903 |
| 정보/풀 | `#map-info`(성공률 분해) + `#map-pull`(시그니처 미보유) | game.js:2915 |
| PDA 미니맵 | 같은 dataURL 축소 + GPS 마킹 | game.js:8224 |
| 좌표 튜닝 | **"하네스 스크린샷으로 % 조정"** — 수작업, 지형과 마커가 서로 독립 | 주석 자인 |

문제의 본질: 배경(그림)과 노드(%)가 **서로 다른 좌표계**라, 지형이 바뀌면 마커를 손으로 다시 맞춘다. 항공뷰 전환의 절반은 이 좌표계 통일 작업이다.

## 2. 구현안 3안 비교

| 안 | 내용 | 판정 |
|---|---|---|
| A. 풀 라이브 3D | 도시 전체를 실시간 항공 씬으로, 줌인도 연속 카메라 | **보류** — 도시 스케일 모델 실시간은 모바일 메모리·프레임 예산 초과 위험, 골든 결정론 부담(#212 계열 확대), 제작량 최대. 콘솔급 목표가 아님 |
| **B. 베이크 파이프라인** | 항공 전도 = 3D 디오라마를 하네스로 **구운 이미지**(도시당 1~2장) · 근접 = 노드당 구운 컷 1장 · 전환 = 크로스줌 | **채택(1차)** — 트레일러/골든 하네스 재사용, 런타임 비용 = 이미지 2장, 결정론 100%, 레퍼런스(사진1→2)와 정확히 같은 체험 |
| C. 하이브리드 | 전도 = 베이크 유지 · **근접만 라이브 3D 미니 씬** (셸터는 기존 `buildEnv` 재사용) | **채택(2차 승격)** — 시간대·날씨·계절이 근접 뷰에 실시간 반영("지금의 그곳"). 셸터 16종은 빌더가 이미 있어 승격 비용이 낮다 |

**B→C 계단**인 이유: B만으로 오더의 체험(정확히 어디를 탐사하는지)이 완성되고, C는 코지 감성(눈 내리는 그 지역을 지도에서 봄)을 더하는 업그레이드라 분리 가능하다.

## 3. 자료구조 (단일 소스 원칙)

### 3.1 노드 테이블 — `src/data/mapnodes.js` (신설, 정본)

```js
// 정본은 "도시 월드좌표"다. 이미지 % 좌표는 베이크가 산출하는 파생물 — 손튜닝 금지.
export const MAP_NODES = {
  // 지역(파밍)
  residential: { kind: 'region', city: 'home', wx: -18, wz: 26 },
  slum:        { kind: 'region', city: 'home', wx: -44, wz: 58,
                 closeup: { cam: { yaw: 0.6, pitch: 0.42, zoom: 2.0 } } }, // 컷 카메라(생략 시 도시 기본)
  // 셸터(거처) — 라이브 승격(C) 시 buildEnv 재사용 지시자
  rooftop:     { kind: 'shelter', city: 'home', wx: -6, wz: 12, live: 'env:rooftop' },
  // 특수(POI) — 응답 불빛·무전 기지 등
  radioBase:   { kind: 'poi', city: 'home', wx: 2, wz: -8, showIf: 'radioBaseDone' },
};
```

### 3.2 베이크 산출 — `src/data/mapnodes.gen.json` (생성물, 수정 금지)

베이크 도구가 월드좌표를 항공 카메라로 **투영해 픽셀 % 좌표를 자동 산출**한다. 현행 "% 수작업 튜닝"이 사라지는 지점 — 지형(디오라마)이 바뀌면 재베이크만 하면 마커가 따라온다.

```json
{ "home": { "residential": { "px": 41.2, "py": 33.8 }, "slum": { "px": 22.7, "py": 63.1 } } }
```

### 3.3 뷰 상태 머신 (모달 내부 상태)

```
city(전도) ──클릭──▶ focus(노드 근접) ──"여기를 탐사"──▶ 기존 탐험 확인 플로우
   ▲                        │
   └────────뒤로/ESC────────┘
state: { level: 'city'|'focus', city, focusId }
```
- `focus`에서 `#map-info`(성공률 분해)·`#map-pull`은 그대로 산다 — 정보 소유권 불변(상세는 PDA, 지도는 판단).
- 도시 탭 전환은 `level:'city'`로 리셋.

### 3.4 에셋 규약 — `public/img/map/`

```
map/home_aerial.png        도시 전도 (2048×1280 근방, 팔레트 양자화)
map/home_aerial_night.png  (선택〔D〕— 밤 전도. 시간대별 분위기)
map/cu_slum.png            노드 근접 컷 (640×360)
map/cu_rooftop.png         …노드당 1장
```
예산 실측 근거: 전도 ~400KB ×2도시(+밤이면 ×2) + 근접 ~80KB ×20노드 ≈ **2~3MB**. APK 147MB 대비 무시 가능. PDA 미니맵은 전도 축소 재사용(추가 비용 0).

## 4. 파일 설계 (관문 원칙 — REFACTOR-GUIDE §2 계승)

| 파일 | 역할 | 패턴 |
|---|---|---|
| `src/data/mapnodes.js` | 노드 정본(월드좌표·게이트·컷 카메라) | SSOT — REGIONS/SHELTERS에서 지도 관심사만 분리 |
| `src/data/mapnodes.gen.json` | 투영 % 좌표 (베이크 산출) | 생성물(glyphs.gen.js 전례) |
| `src/ui/mapview.js` | **openMapModal을 game.js에서 추출**하며 줌 상태 머신 수용 | #73 추출 트랙 계승 — 신규 기능을 God Object에 얹지 않는다 |
| `src/render/dioramas.js` | (C단계) 근접 라이브 미니 씬 — 셸터는 `buildEnv` 어댑터, 지역은 신규 소형 빌더 | Strategy: `live:'env:rooftop'` 지시자 해석 |
| `tools/bake-aerial.mjs` | 도시 디오라마 항공 베이크 + 노드 투영 좌표 산출 + 근접 컷 베이크. `--city`·`--node` 필터(트레일러 clipId 필터 전례) | icon-semiotic/pda-mono 계보 — 코드 생성 에셋 |
| `mapBackdrop(city)` (mapview 내) | 배경 소스 관문: 베이크 있으면 이미지, 없으면 현행 비오메 캔버스 **폴백** | Facade — 저사양/미베이크 도시 안전망. 현행 코드는 삭제가 아니라 강등 |

**카메라 규약**: 항공 전도·근접 컷·(C단계) 라이브 디오라마 전부 **같은 아이소 각**(현 골든 카메라 yaw 0.6 / pitch 0.42 계열)으로 고정한다. 전도→근접→실제 탐험 화면이 같은 시점 문법이라야 "줌인했다"는 착각이 성립한다 — 레퍼런스 사진 1→2가 주는 체험의 핵심.

**디오라마 소스**: 도시 전도용 3D는 새 모델링 대패스가 아니라 **기존 프리팹 재배치**다 — ruin-city 빌딩군(도심 잠식), 셸터 외형(각 buildEnv의 건물), #206 동부 격자(이미 -8° 시가 격자 데이터가 캔버스 코드에 있음 → 배치 데이터로 승격). 아트 신규 제작은 "빈 땅 메우기" 수준.

## 5. 단계 플랜 (공수 추정)

| 단계 | 내용 | 공수 | 산출 체험 |
|---|---|---|---|
| **P0** | `mapnodes.js` 통일(월드좌표) + `bake-aerial.mjs` + 서부 항공 디오라마 조립·베이크 1장 + `mapBackdrop` 관문 교체(핀 로직 그대로) | 1~2세션 | 지도가 "그림"에서 "우리 도시의 항공 사진"이 된다 |
| **P1** | 노드 클릭 → 근접 컷 크로스줌(사진1→2) + 뒤로/브레드크럼 + 근접 컷 전 노드 베이크 | 1세션 | **오더의 핵심 체험 완성** |
| **P2 (=C)** | 근접을 라이브 디오라마로 승격 — 셸터 16종(buildEnv 재사용) 먼저, 지역 12종(신규 소형) 다음. 시간대·날씨 반영 | 2~3세션 | "지금 눈 내리는 그곳"을 지도에서 봄 |
| **P3** | 동부 도시 전도 + PDA 미니맵 승계 + 데모 반영 여부〔D〕 | 1세션 | 2.0 도시뷰와 합류 |

각 단계 독립 발행 가능 — P0만으로도 시각 승급, 중단해도 폴백(비오메 캔버스)이 살아 있다.

## 6. 검증 계획

- **베이크 결정론**: bake-aerial은 시드 고정 + freezeForGolden 문법 — 같은 입력 = 같은 PNG(diff 0). 재베이크가 곧 회귀 테스트.
- **골든 확장**: 지도 모달(전도)·focus(근접) 2씬을 골든에 추가. 단 **라이브 디오라마(P2)는 #212 크로스프로세스 함정 지대** — lodge 플레이키와 같은 부류가 재발할 수 있으니, P2 착수 전에 REFACTOR-GUIDE **P1(렌더 컨텍스트 객체)** 선행을 강권한다. 정적 베이크(P0·P1)는 이 리스크가 0이라 계단 순서가 안전하다.
- **좌표 정합**: 투영 산출 % vs 실제 클릭 히트 — 배터리에 "전 노드 px∈안전영역 + 노드 간 최소 간격" 게이트 추가.
- **모바일**: 2048 텍스처 1장 + 640 컷 1장 동시 로드가 최대치(레벨 전환 시 이전 해제). 위젯 모드는 지도 자체가 안 뜨므로 무관.

## 7. 디렉터 결정 대기

1. **밤 전도 별도 베이크** — 분위기(레퍼런스가 밤 지도) vs 에셋 2배. 권고: 전도는 **밤 고정 1장**(레퍼런스 무드가 정체성, 시간대는 P2 근접 라이브가 담당).
2. **근접 뷰의 최종 형태** — 베이크 컷 유지(가볍고 균질) vs 라이브 승격(P2, 시간대 반영). 권고: P1 출하 후 체감으로 결정.
3. **줌 단계 수** — 2단(전도↔근접, 권고) vs 3단(대륙/도시/근접 — 과설계 위험).
4. **데모 포함 여부** — 데모는 3지역이라 효과 대비 재베이크 비용 소소. 넥페 임팩트용으로 포함 권고.
5. 지역 근접 디오라마의 아트 방향(폐허 밀도·랜드마크) — P2 착수 시 시안으로.
