## 1. 시안 4종 판독

| 코드 | 별칭 | 골자 |
|---|---|---|
| **A** | 지휘 콘솔 | 좌측 3단 패널(거점 스펙·생존 상태·임무 목표) + 우상단 상태바 + 우측 통신 시스템 + 하단 액션바(F1~F5) |
| **B** | 필드 HUD | 좌상단 거처 요약(쾌적+게이지 3) · 우상단 TIME/WEATHER/WIND · 우측 RADIO · 좌하단 ENVIRONMENT · 하단 번호 액션 6 |
| **C** | PDA 중심 | 실물 PDA를 화면 중앙에 크게 배치, LCD 안에 상태·거처·명령 전부. 좌측 보조 패널 + 우측 PDA/채널 탭 |
| **D** | 터미널 정통 | PDA LCD가 진짜 텍스트 터미널 — 블록문자 게이지(■■■□□), 시스템 로그(타임스탬프), 번호 명령(01~04), 푸터 키 안내 |

**공통 언어**: 인광 그린 단색, 얇은 프레임 + 코너 브래킷, `//` 섹션 프리픽스, 모노스페이스 수치. → **이미 v1.9.9 노스트로모 전환으로 우리가 쓰는 문법**이다. 즉 시안은 "새 방향"이 아니라 **현 방향의 완성형 목표치**다. 이게 이 검토의 가장 중요한 판정이다.

## 2. 즉시 채택 가치 (저비용 · 고효과)

현 시스템만으로 구현 가능하고, 신규 시스템을 요구하지 않는 것들.

1. **블록문자 게이지 (D)** — `■■■■■□□□` 텍스트 게이지. 둥근모꼴 글리프만 쓰므로 아트 0, 어느 배율에서도 안 깨짐. v1.9.9 디지털 세그먼트 게이지의 **텍스트판**이라 문법 충돌 없음. 다만 i18n 폭(ko/en/ja 자간)과 스크린리더 대응은 검증 필요.
2. **시스템 로그 (D)** — `[07:41] 경고: 에너지 수치 감소` 형식의 타임스탬프 로그. **게임이 이미 이 데이터를 만든다**(토스트·dayLog.notes·인카운터 결과). 지금은 토스트가 3초 뒤 증발해 "놓치면 끝"인데, 로그는 남는다 — UX 개선 + 컨셉 정합이 동시에 잡히는 드문 항목. **가장 추천.**
3. **번호/단축키 배지 (A·B·D)** — 액션 버튼에 `F1`·`1`·`01` 병기. #210 키바인딩이 이미 있으므로 라벨 노출만 하면 된다. 터미널 감성 + 접근성 + 조작 속도 3중 이득.
4. **패널 내부 셀 그리드 (A)** — 값을 `라벨/수치` 셀로 묶어 격자에 앉히는 조판. 현 패널은 줄글에 가까워 스캔이 느리다. 순수 CSS 작업.
5. **푸터 키 안내 (D)** — `ENTER:실행 TAB:다음탭 Q:종료`. 콘솔 실감 + 온보딩 보조.

## 3. 위험 · 충돌 (채택 전 반드시 판단)

### 3.1 시안 A는 **다른 장르의 게임**을 그리고 있다 — 최대 위험
A의 데이터가 요구하는 것: `거점 레벨`, `방어 등급 B`, `인원 5/6`, `수용 가능 +2`, `산소 여유 72%`, `시설 내구도`, `정비` 액션, `통신 채널 12 / 수신 감도 / 신호 스캔`.
→ 이건 **승무원 관리 + 기지 방어 시뮬**(폴아웃 쉘터/프로스트펑크)의 지표군이다. 나인윈터스는 **혼자 + 고양이**이고, 세계관 정본이 「둘러싸였으나 혼자」다([[worldview-canon]]). `인원 5/6`을 넣는 순간 정본이 깨지고, 방어 등급은 "온스크린 전투 금지" 규칙과 충돌한다.
**판정: A의 데이터 모델은 기각. 레이아웃 문법(셀 그리드·패널 위계)만 추출.**

### 3.2 상시 패널 4~6개 = #211이 해결한 문제의 재발
#211에서 필드노트를 폐지하고 PDA로 통합한 근거는 *"두 기기가 기록·자원·날씨를 중복 표시, 진입문 3개"* 였다. 시안들은 상시 패널을 4~6개 세운다 — 잘못 옮기면 그 중복이 부활한다.
**완화**: 패널을 늘리되 **소유권을 1개씩만** 부여(상태=여기, 물자=PDA, 기록=로그). 같은 수치를 두 곳에 그리지 않는다는 규칙을 먼저 세우고 착수.

### 3.3 시안 C는 코어 루프를 가린다
C는 PDA가 화면의 ~40%를 점유한다. 나인윈터스의 코어는 **내가 꾸민 방을 바라보는 것**인데, 기기가 그 위를 덮는다. 현 설계(PDA는 가장자리 도크, 열면 오버레이)가 코어 루프에 더 맞다.
**판정: C의 "PDA 상시 대형 배치"는 기각. 단, C의 LCD 내부 조판(상태→거처→명령 3단)은 현 PDA 앱에 그대로 이식 가능 — 그건 채택 후보.**

### 3.4 모바일/위젯 미대응
시안은 전부 16:9 데스크톱 와이드다. 우리는 APK(Capacitor)와 위젯 모드를 굽는다. 네 귀퉁이 패널을 폰에 그대로 올리면 화면이 패널로 덮인다.
**필수 선결**: 각 패널의 모바일 거동(접힘/탭 이동/숨김)을 시안 단계에서 정해야 한다. 안 정하면 데스크톱만 예쁘고 폰은 깨진다.

### 3.5 게이지 색 규칙이 현행과 충돌
v1.9.10에서 **디렉터 목업대로** 게이지별 고유색을 넣었다(허기 머스터드·갈증 시안·에너지 그린·청결 라임 = 색이 *정체*를 표시). 반면 시안 B/D는 **색이 심각도**를 표시한다(정상 그린 → 경고 앰버 → 위험 적색).
두 규칙은 동시에 성립하지 않는다. 세미오틱 글리프는 후자(색=상태)를 택했으므로, 리워크로 가면 **v1.9.10 고유색을 되돌리는 판단**이 필요하다.
**결정 필요 항목 1순위.**

### 3.6 신규 시스템을 은근히 요구하는 지표들
- `체감 온도 -21°C`, `WIND 12KM/H NW`, `동상 위험: 낮음` (B) → 풍속·체감온도·동상은 **없는 시스템**. 한파 심각도를 온도풍으로 *표기*만 하는 건 저비용, 실제 시스템화는 별건 대형.
- `협력 채널 / 신호 강도 / 채널 변경` (B·C) → 라디오는 존재하나 채널 시스템은 없음.
**원칙: 표기 리스킨(싸다) vs 시스템 신설(비싸다)을 항목마다 명시적으로 갈라서 결정한다.**

## 5. 디렉터 결정 필요

1. **게이지 색 규칙** — 색=정체(현 v1.9.10) vs 색=심각도(시안). 리워크 가면 후자로 통일이 자연스러움.
2. **시안 A 데이터 모델** — 승무원·산소·방어등급은 정본 충돌. 기각 확인 필요(아니면 세계관 개정이 선행돼야 함).
3. **PDA 배치** — 가장자리 도크 유지(현행·추천) vs 중앙 상시(C안).
4. **착수 시점** — 데모(10월) 전 P0~P1만 vs 데모 후 P3까지.
5. **모바일 거동** — 리워크 시안에 폰 레이아웃을 포함시킬지(권장) 여부.

## 6. 총평

시안들은 **우리가 이미 가고 있는 방향의 완성형**이다 — v1.9.9 노스트로모 전환, v1.9.10 탈아이콘, v1.9.11 세미오틱 글리프가 향하던 지점을 그림으로 못 박은 것에 가깝다. 그래서 "방향 전환"이 아니라 **"조판 밀도와 정보 위계를 한 단계 올리는 일"** 로 다루는 게 맞다.

다만 시안에 섞여 들어온 **장르 이탈 지표(승무원·방어·산소)** 는 분리해서 버려야 한다. 그림이 예뻐 보이는 이유의 일부가 "정보가 많아서"인데, 그 정보 중 일부는 우리 게임에 존재하지 않으며 만들면 게임이 다른 게임이 된다. **조판은 가져오고, 데이터 모델은 가져오지 않는다** — 이 한 줄이 이 검토의 결론이다.
